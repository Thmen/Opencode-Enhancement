# ClaudeCode 编号稳定性修复设计

## 背景

`scripts/claudecode_scraper_core.py` 当前在 `_build_pages()` 中采用“仅对成功抓取页面重新连续编号”的策略：

- 先遍历 `records`
- 失败页面直接跳过
- 对成功页面按分类重新累计 `counters`

这会导致一个问题：只要某个页面抓取失败，或者在失败页之前插入一个新页，后续同分类页面的文件名前缀就会整体漂移。

## 目标

- 保持 `claudecode` 文档编号对 `records` 原始顺序稳定
- 即使某页抓取失败，也保留其编号空位，不让后续页面补位
- 不改变现有目录结构、链接本地化和 README 生成方式

## 非目标

- 不改 `opencode` 抓取器
- 不解决上游中文站缺页问题
- 不改变分类规则或页面发现来源

## 方案

在 `_build_pages(records, config)` 中，把“编号分配”从“成功抓取后再连续计数”改成“按 `records` 顺序预分配固定编号”。

### 具体规则

- 编号仍然按分类分别计数
- 但计数依据是 `records` 的原始遍历顺序，而不是抓取成功的子集
- 某条记录即使抓取失败，它所属分类的计数也要继续前进
- 成功页面写盘时直接使用为该记录预留的固定编号

## 示例

若根目录记录依次是：

1. `channels`
2. `changelog`
3. `checkpointing`

并且本次 `changelog` 失败：

- `channels` 仍是 `01-*`
- `changelog` 没有文件，但占用 `02`
- `checkpointing` 必须仍是 `03-*`

而不是错误地变成 `02-*`。

## 测试设计

新增失败测试，模拟：

- 同分类前一页失败，后一页成功
- 断言后一页仍然拿到原始应有编号，而不是补位编号

至少覆盖：

- 顶层未分类页面
- 分类内页面

## 风险与控制

- 风险：README 中出现编号空洞
  - 这是期望行为，因为目标就是保留空位而不是重排
- 风险：后续新增页面改变既有编号
  - 这只会影响它之后的页面；但这是基于源顺序的真实结构变化，不是“失败导致的伪漂移”
