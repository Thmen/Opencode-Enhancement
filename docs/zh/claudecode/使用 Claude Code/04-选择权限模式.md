# 选择权限模式

> 在监督编辑、只读规划和自动模式之间切换，其中后台分类器替代手动权限提示。在 CLI 中使用 Shift+Tab 循环切换模式，或在 VS Code、Desktop 和 claude.ai 中使用模式选择器。

权限模式控制 Claude 是否在执行操作前询问。不同的任务需要不同级别的自主权：您可能希望对敏感工作进行完全监督，对长时间重构进行最少中断，或在探索代码库时进行只读访问。

本页面涵盖以下内容：

* [切换模式](../快速开始/01-ClaudeCode概述.md#switch-permission-modes)：在会话期间、启动时或作为默认设置
* [选择模式](../快速开始/01-ClaudeCode概述.md#available-modes)：基于 Claude 无需询问即可执行的操作
* [运行自动模式](../快速开始/01-ClaudeCode概述.md#eliminate-prompts-with-auto-mode)：使用后台安全检查，并查看它[默认阻止的内容](../快速开始/01-ClaudeCode概述.md#what-the-classifier-blocks-by-default)
* [以只读方式规划更改](../快速开始/01-ClaudeCode概述.md#analyze-before-you-edit-with-plan-mode)：在批准编辑前进行分析
* [将 Claude 限制为预先批准的工具](../快速开始/01-ClaudeCode概述.md#allow-only-pre-approved-tools-with-dontask-mode)：用于锁定环境
* [完全跳过检查](../快速开始/01-ClaudeCode概述.md#skip-all-checks-with-bypasspermissions-mode)：在隔离环境中

## 切换权限模式

您可以在会话期间、启动时或作为持久默认设置随时切换模式。具体机制取决于您在哪里运行 Claude Code。

#### CLI
**在会话期间**：按 `Shift+Tab` 循环切换 `default` → `acceptEdits` → `plan` → `auto`。当前模式显示在状态栏中。在启动时传递 `--enable-auto-mode` 之前，`auto` 不会出现在循环中。Auto 还需要 Team（或 Enterprise/API 即将推出）计划和 Claude Sonnet 4.6 或 Opus 4.6，因此即使使用该标志，该选项也可能保持不可用。如果同时启用了 `bypassPermissions`，它会出现在 `plan` 和 `auto` 之间的循环中。

**启动时**：将模式作为 CLI 标志传递：

```bash
    claude --permission-mode plan
    ```

**作为默认设置**：在您的[设置文件](../45-ClaudeCode设置.md#settings-files)中设置 `defaultMode`：

```json
    {
      "permissions": {
        "defaultMode": "acceptEdits"
      }
    }
    ```

**非交互式**：相同的标志适用于脚本运行，使用 `-p`：

```bash
    claude -p "refactor auth" --permission-mode acceptEdits
    ```

`dontAsk` 永远不会出现在 `Shift+Tab` 循环中。`bypassPermissions` 仅在您使用 `--permission-mode bypassPermissions`、`--dangerously-skip-permissions` 或 `--allow-dangerously-skip-permissions` 启动会话时才会出现在循环中。第三个标志将模式添加到循环中而不激活它，因此您可以将其与不同的起始模式（如 `--permission-mode plan`）组合。在启动时或在设置文件中设置这些中的任何一个。

#### JetBrains
JetBrains 插件在 IDE 终端中启动 Claude Code，因此切换模式的工作方式与 CLI 中相同：按 `Shift+Tab` 循环切换，或在启动时传递 `--permission-mode`。

#### VS Code
**在会话期间**：单击提示框底部的模式指示器以切换模式。

**作为默认设置**：在 VS Code 设置中设置 `claudeCode.initialPermissionMode`，或使用 Claude Code 扩展设置面板。

VS Code UI 使用友好标签，映射到下面的设置键：

| UI 标签              | 设置键                 |
| :----------------- | :------------------ |
| Ask permissions    | `default`           |
| Auto accept edits  | `acceptEdits`       |
| Plan mode          | `plan`              |
| Auto               | `auto`              |
| Bypass permissions | `bypassPermissions` |

Auto 和 Bypass permissions 仅在您在扩展设置中启用**Allow dangerously skip permissions**后才会出现。Auto 还需要 Team 计划和 Claude Sonnet 4.6 或 Opus 4.6，因此即使打开切换开关，该选项也可能保持不可用。

有关扩展特定的详细信息，请参阅 [VS Code 指南](../平台和集成/06-在VSCode中使用ClaudeCode.md)。

#### Desktop
**在会话期间**：使用发送按钮旁边的模式选择器。您可以在会话前或会话期间更改它。

Desktop UI 使用友好标签，映射到下面的设置键：

| UI 标签              | 设置键                 |
| :----------------- | :------------------ |
| Ask permissions    | `default`           |
| Auto accept edits  | `acceptEdits`       |
| Plan mode          | `plan`              |
| Auto               | `auto`              |
| Bypass permissions | `bypassPermissions` |

Auto 和 Bypass permissions 仅在您在 Desktop 设置中启用它们后才会出现在选择器中。有关详细信息，请参阅 [Desktop 指南](../15-使用ClaudeCodeDesktop.md#choose-a-permission-mode)。

#### Web and mobile
**在会话期间**：在 [claude.ai/code](https://claude.ai/code) 或 Claude 移动应用中使用提示框旁边的模式下拉菜单。

对于在 Anthropic 云 VM 上运行的 [Claude Code on the web](../09-ClaudeCodeontheweb.md) 会话，下拉菜单提供 Auto accept edits 和 Plan mode。Ask permissions 和 Auto 不适用于云会话。

对于在您的本地机器上运行的 [Remote Control](../平台和集成/04-使用RemoteControl从任何设备继续本地会话.md) 会话，下拉菜单提供 Ask permissions、Auto accept edits 和 Plan mode。您还可以在启动本地主机时设置起始模式：

```bash
    claude remote-control --permission-mode acceptEdits
    ```

权限提示出现在 claude.ai 中以供批准。

权限模式通过 UI、CLI 标志或设置文件设置。在聊天中告诉 Claude"停止要求权限"不会改变模式。有关模式如何与允许、询问和拒绝规则交互的信息，请参阅[权限](../37-配置权限.md)。

## 可用模式

每种模式在便利性和监督之间做出不同的权衡。选择与您的任务相匹配的模式。

| 模式                                                                  | Claude 无需询问即可执行的操作 | 最适合             |
| :------------------------------------------------------------------ | :----------------- | :-------------- |
| `default`                                                           | 读取文件               | 入门、敏感工作         |
| `acceptEdits`                                                       | 读取和编辑文件            | 迭代您正在审查的代码      |
| [`plan`](../快速开始/01-ClaudeCode概述.md#analyze-before-you-edit-with-plan-mode)                   | 读取文件               | 探索代码库、规划重构      |
| [`auto`](../快速开始/01-ClaudeCode概述.md#eliminate-prompts-with-auto-mode)                         | 所有操作，带后台安全检查       | 长时间运行的任务、减少提示疲劳 |
| [`bypassPermissions`](../快速开始/01-ClaudeCode概述.md#skip-all-checks-with-bypasspermissions-mode) | 所有操作，无检查           | 仅隔离容器和 VM       |
| [`dontAsk`](../快速开始/01-ClaudeCode概述.md#allow-only-pre-approved-tools-with-dontask-mode)       | 仅预先批准的工具           | 锁定环境            |

## 使用 Plan Mode 在编辑前进行分析

Plan mode 告诉 Claude 研究并提议更改而不进行更改。Claude 读取文件、运行 shell 命令进行探索、提出澄清问题并编写计划文件，但不编辑您的源代码。权限提示的工作方式与默认模式相同：您仍然批准 Bash 命令、网络请求和其他通常会提示的操作。

### 何时使用 Plan mode

当您希望 Claude 在进行更改前研究并提议方法时，Plan mode 很有用：

* **多步骤实现**：当功能需要跨多个文件的编辑时
* **代码探索**：当您想在更改任何内容前研究代码库时
* **交互式开发**：当您想与 Claude 迭代方向时

### 启动和使用 Plan mode

通过在提示前加上 `/plan` 为单个请求进入 Plan mode，或通过按 `Shift+Tab` [循环切换权限模式](../快速开始/01-ClaudeCode概述.md#switch-permission-modes)将整个会话切换到 Plan mode。您也可以从 CLI 启动 Plan mode：

```bash
claude --permission-mode plan
```

此示例为复杂重构启动规划会话：

```text
I need to refactor our authentication system to use OAuth2. Create a detailed migration plan.
```

Claude 分析当前实现并创建计划。使用后续问题进行细化：

```text
What about backward compatibility?
How should we handle database migration?
```

当计划准备好时，Claude 呈现它并询问如何继续。从该提示您可以：

* 批准并在自动模式下启动
* 批准并接受编辑
* 批准并手动审查每个编辑
* 继续规划，将您的反馈发送回 Claude 进行另一轮

每个批准选项也提供首先清除规划上下文的选项。

## 使用自动模式消除提示

Auto mode 在 Team 计划上可用，Enterprise 和 API 支持即将推出。在 Team 和 Enterprise 上，管理员必须在 [Claude Code 管理员设置](https://claude.ai/admin-settings/claude-code)中启用它，然后用户才能打开它。它需要 Claude Sonnet 4.6 或 Claude Opus 4.6，在 Haiku、claude-3 模型或第三方提供商（Bedrock、Vertex、Foundry）上不可用。

Auto mode 让 Claude 执行操作而不显示权限提示。在每个操作运行前，一个单独的分类器模型审查对话并决定该操作是否与您要求的相匹配：它阻止超出任务范围的操作、针对分类器不认识为受信任的基础设施的操作，或似乎由文件或网页中遇到的恶意内容驱动的操作。有关分类器设计方式的更深入了解，请参阅 [auto mode 公告](https://claude.com/blog/auto-mode)。

**Warning:**
Auto mode 是研究预览版。它减少提示但不保证安全。它提供比 `bypassPermissions` 更多的保护，但不如手动审查每个操作那样彻底。将其用于您信任一般方向的任务，而不是作为敏感操作审查的替代品。

**模型**：分类器在 Claude Sonnet 4.6 上运行，即使您的主会话使用不同的模型。

**成本**：分类器调用计入您的令牌使用，与主会话调用相同。每个检查的操作向分类器发送对话记录的一部分加上待处理操作。额外成本主要来自 shell 命令和网络操作，因为只读操作和工作目录中的文件编辑不会触发分类器调用。

**延迟**：每个分类器检查在操作执行前添加一个往返。

### 操作如何被评估

每个操作都经过固定的决策顺序。第一个匹配的步骤获胜：

1. 与您的[允许或拒绝规则](../37-配置权限.md#manage-permissions)匹配的操作立即解决
2. 只读操作和工作目录中的文件编辑自动批准
3. 其他所有内容都发送到分类器
4. 如果分类器阻止，Claude 收到原因并尝试替代方法

进入自动模式时，Claude Code 删除任何已知授予任意代码执行的允许规则：诸如 `Bash(*)` 之类的全面 shell 访问、诸如 `Bash(python*)` 或 `Bash(node*)` 之类的通配符脚本解释器、包管理器运行命令以及任何 `Agent` 允许规则。这些规则会自动批准命令和子代理委派，这些命令和委派最有能力造成损害，然后分类器才能看到它们。诸如 `Bash(npm test)` 之类的狭隘规则会继续。删除的规则在您离开自动模式时恢复。

分类器接收用户消息和工具调用作为输入，Claude 自己的文本和工具结果被剥离。它还接收您的 CLAUDE.md 内容，因此您的项目说明中描述的操作会被纳入允许和阻止决策中。因为工具结果永远不会到达分类器，文件或网页中的恶意内容无法直接操纵它。分类器根据可自定义的阻止和允许规则集评估待处理操作，检查该操作是否是超出您要求的过度升级、关于什么是安全接触的错误，或与您陈述的意图的突然偏离，表明 Claude 可能已被它读到的东西引导。

与您的权限规则不同，权限规则匹配工具名称和参数模式，分类器读取要阻止和允许的内容的散文描述：它根据上下文推理操作而不是匹配语法。

### Auto mode 如何处理子代理

当 Claude 生成[子代理](../49-创建自定义subagents.md)时，分类器在子代理启动前评估委派的任务。看起来本身很危险的任务描述，如"删除与此模式匹配的所有远程分支"，在生成时被阻止。

在子代理内，auto mode 使用与父会话相同的阻止和允许规则运行。子代理在其自己的前言中定义的任何 `permissionMode` 都被忽略。子代理自己的工具调用独立地通过分类器。

当子代理完成时，分类器审查其完整的操作历史。在生成时无害的子代理可能在运行中期被它读到的内容破坏。如果返回检查标记了一个问题，安全警告会被添加到子代理的结果前面，以便主代理可以决定如何继续。

### 分类器默认阻止的内容

开箱即用，分类器信任您的工作目录，如果您在 git 仓库中，该仓库的配置的远程。其他所有内容都被视为外部：您的公司源代码控制组织、云桶和内部服务是未知的，直到您告诉分类器它们。

**默认阻止**：

* 下载和执行代码，如 `curl | bash` 或来自克隆仓库的脚本
* 向外部端点发送敏感数据
* 生产部署和迁移
* 云存储上的大规模删除
* 授予 IAM 或仓库权限
* 修改共享基础设施
* 不可逆地销毁会话开始前存在的文件
* 破坏性源代码控制操作，如强制推送或直接推送到 `main`

**默认允许**：

* 工作目录中的本地文件操作
* 安装已在您的锁定文件或清单中声明的依赖项
* 读取 `.env` 并向其匹配的 API 发送凭证
* 只读 HTTP 请求
* 推送到您启动的分支或 Claude 创建的分支

要查看分类器接收的完整默认规则列表，请运行 `claude auto-mode defaults`。

如果自动模式阻止了对您的团队来说很常规的事情，如推送到您自己的组织仓库或写入公司桶，那是因为分类器不知道这些是受信任的。管理员可以通过 `autoMode.environment` 设置添加受信任的仓库、桶和内部服务：有关完整配置指南，请参阅[配置自动模式分类器](../37-配置权限.md#configure-the-auto-mode-classifier)。

### 当自动模式回退时

回退设计防止误报使会话脱轨：错误的阻止使 Claude 重试，而不是您的进度。如果分类器在一行中阻止操作 3 次或在一个会话中总共 20 次，自动模式暂停，Claude Code 恢复为每个操作提示。这些阈值不可配置。

* **CLI**：您在状态区域看到通知。批准提示的操作重置拒绝计数器，因此您可以继续在自动模式中
* **非交互式模式**（带 `-p` 标志）：中止会话，因为没有用户可以提示

重复的阻止通常意味着两件事之一：任务确实需要分类器构建来停止的操作，或分类器缺少关于您的受信任基础设施的上下文，并将安全操作视为有风险。如果阻止看起来像误报，或者分类器遗漏了它应该捕获的东西，使用 `/feedback` 报告它。如果阻止发生是因为分类器不认识您的仓库或服务为受信任，让管理员在托管设置中[配置受信任基础设施](../37-配置权限.md#configure-the-auto-mode-classifier)。

## 仅使用 dontAsk mode 允许预先批准的工具

`dontAsk` mode 自动拒绝每个未明确允许的工具。只有与您的 `/permissions` 允许规则或 `permissions.allow` 设置匹配的操作才能执行。如果工具有明确的 `ask` 规则，操作也被拒绝而不是提示。这使模式完全非交互式，适合 CI 管道或受限环境，其中您预先定义 Claude 被允许执行的确切操作。

```bash
claude --permission-mode dontAsk
```

## 使用 bypassPermissions mode 跳过所有检查

`bypassPermissions` mode 禁用所有权限提示和安全检查。每个工具调用立即执行而无需任何验证。仅在隔离环境（如容器、VM 或没有互联网访问的 devcontainers）中使用此模式，其中 Claude Code 无法对您的主机系统造成损害。

```bash
claude --permission-mode bypassPermissions
```

`--dangerously-skip-permissions` 标志等同于 `--permission-mode bypassPermissions`：

```bash
claude -p "refactor the auth module" --dangerously-skip-permissions
```

**Warning:**
`bypassPermissions` mode 不提供针对提示注入或意外操作的保护。对于仍然维护后台安全检查的更安全替代品，使用 [auto mode](../快速开始/01-ClaudeCode概述.md#eliminate-prompts-with-auto-mode)。管理员可以通过在[托管设置](../37-配置权限.md#managed-settings)中将 `permissions.disableBypassPermissionsMode` 设置为 `"disable"` 来阻止此模式。

## 比较权限方法

下表总结了每种模式在处理批准方面的关键差异。`plan` 被省略，因为它限制 Claude 可以执行的操作而不是批准的工作方式。

|      | `default` | `acceptEdits` | `auto`     | `dontAsk`     | `bypassPermissions` |
| :--- | :-------- | :------------ | :--------- | :------------ | :------------------ |
| 权限提示 | 文件编辑和命令   | 仅命令           | 无，除非回退触发   | 无，除非预先允许否则被阻止 | 无                   |
| 安全检查 | 您审查每个操作   | 您审查命令         | 分类器审查命令    | 仅您的预先批准规则     | 无                   |
| 令牌使用 | 标准        | 标准            | 更高，来自分类器调用 | 标准            | 标准                  |

## 进一步自定义权限

权限模式设置基线批准行为。对于对单个工具或命令的控制，在活动模式之上分层额外配置。

**权限规则**是第一站。将 `allow`、`ask` 或 `deny` 条目添加到您的设置文件以预先批准安全命令、强制对有风险的命令进行提示或完全阻止特定工具。规则在除 `bypassPermissions` 外的每种模式中应用，`bypassPermissions` 完全跳过权限层，并通过工具名称和参数模式匹配。有关语法和示例，请参阅[管理权限](../37-配置权限.md#manage-permissions)。

**Hooks** 涵盖模式匹配规则无法表达的逻辑。[`PreToolUse` hook](../25-Hooks参考.md#pretooluse-decision-control) 在每个工具调用前运行，可以根据命令内容、文件路径、时间或来自外部策略服务的响应允许、拒绝或升级。[`PermissionRequest` hook](../25-Hooks参考.md#permissionrequest) 拦截权限对话本身并代表您回答。有关配置，请参阅 [Hooks](../25-Hooks参考.md)。
